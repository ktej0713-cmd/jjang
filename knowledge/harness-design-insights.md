## 1. 핵심 개념 요약

### 하네스(Harness)란?
AI 에이전트가 장시간 자율적으로 작동할 수 있도록 감싸는 **구조적 틀**.
단순 프롬프트가 아니라 에이전트의 "환경 설계"를 뜻한다.

### 두 가지 핵심 문제

| 문제 | 설명 | 비유 |
|------|------|------|
| **컨텍스트 불안** | 대화가 길어지면 AI가 남은 일을 대충 마무리하려 함 | 퇴근 시간 다가오면 대충 처리하는 직원 |
| **자기 평가 편향** | AI가 자기 결과물을 항상 "잘했다"고 평가 | 내가 쓴 글을 내가 읽으면 잘 쓴 것 같은 현상 |

### GAN 영감의 3-에이전트 구조

| 역할 | 기능 | 핵심 원칙 |
|------|------|----------|
| **Planner** | 한 줄 요청 → 전체 제품 스펙 확장 | **무엇**만 정하고, **어떻게**는 구현자에게 위임 |
| **Generator** | 스프린트 단위로 기능 구현 | 스프린트 전 Evaluator와 완료 기준 합의 |
| **Evaluator** | Playwright로 실제 브라우저 테스트 + 4기준 점수화 | 생성자와 반드시 분리, 구체적 패스/페일 기준 |

### Evaluator의 4가지 평가 기준
1. **디자인 품질** (가중치 높음)
2. **독창성** (가중치 높음) - "AI 슬롭 패턴" 명시적 감점
3. **완성도**
4. **기능성**

### Opus 4.6 이후 단순화
- 컨텍스트 불안 거의 해소 → 스프린트 분할 불필요
- 자동 compaction 활용 → 컨텍스트 리셋 로직 제거
- 평가를 최종 1회로 축소
- **결과**: 6시간/$200 → 3시간 50분/$124 (비용 38% 절감)

### 가장 중요한 문장
> "모델이 좋아져도 하네스 조합의 가능성은 줄어들지 않는다. 이동할 뿐이다."
> → 모델 발전 시 기존 스캐폴딩 일부는 제거하되, 더 높은 수준의 새로운 구조가 필요해진다.

## 2. jjang-md-team 현재 구조 vs 하네스 패턴 매핑

| 하네스 패턴 | jjang-md-team 현재 | 일치도 | 개선점 |
|------------|-------------------|--------|--------|
| Planner | Lead MD | 높음 | 세부 구현 지시 줄이기 |
| Generator | 소싱/콘텐츠/구조/프로모션 MD | 높음 | 스프린트 전 완료 기준 합의 추가 |
| Evaluator | 고객검증 MD | 높음 | 구체적 패스/페일 체크리스트 강화 |
| 생성-평가 분리 | 이미 분리됨 | 완벽 | 유지 |
| Playwright 테스트 | 테스터 에이전트 | 부분 | HTML 실제 렌더링 검증 추가 |

**결론: 현재 구조가 Anthropic 하네스 패턴과 이미 높은 수준으로 일치한다.**
아래 5가지 개선으로 완성도를 높일 수 있다.

## 3. 구체적 개선안 (5가지)

### 개선 1: Lead MD의 "무엇만 정하기" 원칙 강화

**현재**: Lead MD가 때때로 HTML 구조, 섹션 순서까지 지정
**개선**: 방향과 목표만 전달, 구현 방법은 각 MD에게 위임

```
[AS-IS] Lead MD 브리핑
"히어로 섹션에 36px 타이틀, 그 아래 스펙 테이블 3열로..."

[TO-BE] Lead MD 브리핑
"미즈노 프로 글러브 상세페이지. 타겟: 사회인야구 30대.
 핵심 전달: HAGA JAPAN 장인 정신 + 실사용 체감.
 경쟁 차별점: 프로오더 시스템 설명."
```

**Why**: 기획 단계에서 기술 디테일을 틀리면 실수가 전체 구현에 퍼진다.

---

### 개선 2: 고객검증 MD에 하드 패스/페일 체크리스트 도입

**현재**: "야구인 관점에서 PASS/FAIL" (주관적)
**개선**: 구체적이고 측정 가능한 기준 도입

```markdown
## 고객검증 MD 평가 체크리스트 (상세페이지)

### HARD FAIL (하나라도 실패 시 무조건 FAIL)
- [ ] 야구 용어 오류 (예: "내야수용"을 "야수용"으로 표기)
- [ ] 스펙 수치 오류 (사이즈, 무게, 소재)
- [ ] AI 슬롭 디자인 (보라색 그라데이션, 무의미한 아이콘 장식)
- [ ] 가격 하드코딩
- [ ] 고도몰/스마트스토어 비호환 HTML (style태그, class, flex 등)

### SOFT CRITERIA (3개 이상 미달 시 FAIL)
- [ ] 타겟 고객(사회인/유소년/엘리트)에 맞는 톤
- [ ] "야구 해본 사람이 쓴 글" 느낌
- [ ] 스펙 나열 < 체감/용도/상황 설명
- [ ] 사이즈/착용 가이드 포함
- [ ] 비교 정보 (같은 카테고리 내)
- [ ] 모바일 가독성 (860px 이내)
```

---

### 개선 3: 스프린트 전 "계약" 패턴 도입

**Anthropic 인사이트**: Generator와 Evaluator가 작업 전에 완료 기준을 합의한다.

**적용**: 각 MD가 작업 시작 전 Lead MD에게 plan 제출 시, 고객검증 MD의 평가 기준도 함께 명시

```markdown
## 작업 계약서 (상세페이지 예시)

### 작업 목표
미즈노 프로 셀렉트 내야수용 글러브 상세페이지

### 완료 기준 (고객검증 MD와 합의)
1. HAGA JAPAN 설명이 팩트 기반으로 정확할 것
2. 사회인 선수 관점의 체감 설명 3개 이상
3. 경쟁 제품(윌슨 A2K) 대비 차별점 명시
4. 모바일 860px 기준 렌더링 정상
5. 스마트스토어 호환 HTML (인라인 스타일만)

### 평가 방법
- 고객검증 MD가 위 5개 기준으로 PASS/FAIL 판정
- FAIL 시 구체적 수정 포인트 명시 → 재작업 1회
```

---

### 개선 4: 스캐폴딩 단순화 (Opus 4.6 기준)

**Anthropic 교훈**: 모델이 개선되면 불필요한 구조를 제거해야 한다.

| 현재 구조 | 변경 | 이유 |
|----------|------|------|
| 스프린트 분할 작업 | 제거 가능 | Opus 4.6은 컨텍스트 불안 없음, 장시간 일관 작업 가능 |
| 매 단계 Lead MD 중간 검토 | 최종 1회 검토로 축소 | 병렬 작업 후 통합 시점에만 검토 |
| 컨텍스트 리셋 로직 | Claude Agent SDK auto-compaction 활용 | 별도 구현 불필요 |
| 모든 MD 순차 보고 | 병렬 완료 후 일괄 보고 | 대기 시간 절감 |

---

### 개선 5: Evaluator 실제 사용 테스트 강화

**Anthropic 방식**: Playwright로 실제 브라우저에서 클릭/스크롤/스크린샷
**짱베이스볼 적용**: 테스터 에이전트가 생성된 HTML을 실제로 검증

```markdown
## 테스터 검증 항목 (HTML 결과물)

### 렌더링 테스트
- 브라우저에서 열었을 때 레이아웃 깨짐 없는지
- 모바일 뷰(360px)에서 가로 스크롤 없는지
- 이미지 플레이스홀더가 정상 표시되는지

### 호환성 테스트
- style 태그 없이 인라인만으로 구성되었는지
- class 속성 사용하지 않았는지
- flex/grid 대신 table 레이아웃인지
- details/summary 태그 미사용

### 콘텐츠 무결성
- 빈 섹션이나 placeholder 텍스트 남아있지 않은지
- 링크가 깨지지 않았는지
- 한글 인코딩 정상인지
```
